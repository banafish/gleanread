# 规范：android-supabase-auth

## 目的
定义 Android 端的 Supabase 登录能力及数据归属选择策略。

## 需求

### 需求:系统必须支持邮箱密码注册
系统必须在 Android App 中提供 Supabase 邮箱密码注册能力，注册成功后根据 Supabase 配置决定是否需要邮箱验证。

#### 场景:注册成功且无需邮箱验证
- **当** 用户使用邮箱和密码完成注册且 Supabase 未开启邮箱验证
- **那么** 系统必须自动完成登录并持久化会话
- **那么** 系统必须返回设置页面并显示已登录状态

#### 场景:注册成功但需要邮箱验证
- **当** 用户使用邮箱和密码完成注册且 Supabase 开启了邮箱验证
- **那么** 系统必须提示用户查收验证邮件
- **那么** 系统禁止在邮箱验证完成前允许登录

#### 场景:注册失败（邮箱已存在）
- **当** 用户使用已注册的邮箱尝试注册
- **那么** 系统必须显示邮箱已注册的错误提示
- **那么** 系统必须保持当前注册表单状态不变

#### 场景:注册失败（密码不符合要求）
- **当** 用户输入的密码不符合 Supabase 密码策略
- **那么** 系统必须显示密码要求的错误提示

### 需求:注册页面必须验证确认密码
系统必须在注册模式下要求用户输入两次密码，且两次密码必须一致方可提交注册。

#### 场景:密码不一致
- **当** 用户输入的密码与确认密码不一致
- **那么** 系统必须在用户点击注册按钮时显示密码不一致的错误提示
- **那么** 系统禁止发送注册请求

#### 场景:密码一致
- **当** 用户输入的密码与确认密码一致
- **那么** 系统必须允许发送注册请求

### 需求:Android 端必须支持 Supabase 登录会话
系统必须在 Android App 中提供 Supabase 邮箱密码登录能力，并在登录成功后持久化当前会话，使后续云端同步可以使用 Supabase 用户 ID 作为数据归属标识。

#### 场景:未登录时继续本地使用
- **当** 用户尚未登录 Supabase
- **那么** 系统必须允许用户继续使用本地工作台读写数据
- **那么** 系统必须禁止执行云端同步上传或拉取

#### 场景:登录成功后保存会话
- **当** 用户使用邮箱和密码完成 Supabase 登录
- **那么** 系统必须持久化登录会话
- **那么** 系统必须能够读取当前 Supabase 用户 ID 供同步层写入 `user_id`

#### 场景:登录失败
- **当** 用户输入的邮箱或密码无法完成 Supabase 登录
- **那么** 系统必须保留当前未登录或原会话状态
- **那么** 系统必须显示登录失败原因
- **那么** 系统禁止执行云端同步上传或拉取

#### 场景:会话失效
- **当** 已登录会话失效或刷新失败
- **那么** 系统必须停止云端同步
- **那么** 系统必须保留本地数据，并提示用户重新登录后再继续同步

#### 场景:登录/注册入口从独立页面访问
- **当** 未登录用户需要登录或注册
- **那么** 系统必须通过独立的全屏登录/注册页面提供入口
- **那么** 系统禁止在设置页面内嵌登录表单
- **那么** 登录/注册页面必须支持在登录模式和注册模式之间切换

### 需求:登录后必须由用户选择本地数据归属
当用户登录 Supabase 且本地存在 `local-user` 工作台数据时，系统必须让用户显式选择数据归属，禁止在用户选择前自动上传、覆盖或删除本地数据。

#### 场景:检测到本地数据
- **当** 用户登录成功且本地存在 `user_id = local-user` 的摘录、节点、标签或关系数据
- **那么** 系统必须显示数据归属选择
- **那么** 系统必须在用户选择前暂停云端同步

#### 场景:合并本地数据到当前账号
- **当** 用户选择将本地数据合并到当前 Supabase 账号
- **那么** 系统必须保留本地记录的 `id`
- **那么** 系统必须把相关记录的 `user_id` 更新为当前 Supabase 用户 ID
- **那么** 系统必须把相关记录标记为待上传状态，并写入当前 `device_id` 与 `local_dirty_time`

#### 场景:保留本地模式
- **当** 用户选择保留本地模式
- **那么** 系统必须保留 `local-user` 数据不变
- **那么** 系统必须保持云端同步关闭

#### 场景:使用云端数据替换本地数据
- **当** 用户选择使用当前账号的云端数据
- **那么** 系统必须先清空本地工作台数据
- **那么** 系统必须从 Supabase 拉取当前用户的云端数据并写入 Room

### 需求:退出登录必须停止同步且不破坏本地数据
系统必须支持用户退出 Supabase 登录。在退出时，系统必须先拦截并检查同步队列状态；退出后停止所有网络同步并安全关闭对应的数据库连接，但不删除对应的 `.db` 文件，同时将状态重置为使用独立的访客数据库。

#### 场景:退出登录时的队列拦截与清理
- **当** 用户退出 Supabase 登录
- **那么** 系统必须先检查当前用户的同步队列中是否存在未上传的本地修改，若有则必须弹窗警告用户
- **那么** 系统必须清除本地 Supabase 会话并断开后台同步及 Realtime 订阅
- **那么** 系统必须关闭当前用户的 `local_data_{user_id}.db` 数据库连接，且禁止删除该文件
- **那么** 系统必须清空 Activity 栈并将用户重置至访客状态（重新绑定至 `guest.db`）
