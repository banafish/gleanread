# 规范：android-supabase-cloud-sync

## 目的
定义 Android 端与 Supabase 云端的双向同步、状态管理、冲突处理和实时更新机制。

## 需求

### 需求:Supabase 云端表必须镜像核心知识库数据
系统必须在 Supabase Postgres 中为 `knowledge_tree_node`、`tags`、`excerpts`、`excerpt_tags` 建立与 Android 核心数据对应的云端表，并为每张表保存 `id`、`user_id`、`create_time`、`update_time`、`is_deleted`、`device_id` 以及各自的业务字段。

#### 场景:写入云端记录
- **当** Android 同步器向 Supabase 上传任一核心表记录
- **那么** 云端记录必须保存该记录的主键、用户归属、创建时间、修改时间、软删除状态和最后修改设备
- **那么** 云端记录禁止保存本地专用的 `sync_status`、`last_sync_time`、`sync_error`、`retry_count`、`local_dirty_time`

#### 场景:跨用户隔离
- **当** 已登录用户访问 Supabase 云端表
- **那么** Supabase RLS 必须只允许该用户读取和写入 `user_id` 等于当前 Supabase 用户 ID 的记录
- **那么** 该用户禁止读取或写入其他用户的数据

### 需求:本地变更必须异步上传到 Supabase
系统必须通过同步引擎扫描本地待同步记录并上传到 Supabase，上传过程不得阻塞用户的本地读写操作。

#### 场景:上传待创建记录
- **当** 本地记录的 `sync_status` 为 `PENDING_CREATE`
- **那么** 同步器必须把该记录上传到对应的 Supabase 表
- **那么** 上传期间必须把该记录本地状态切换为 `SYNCING`
- **那么** 上传成功后必须把该记录本地状态切换为 `SYNCED` 并更新 `last_sync_time`

#### 场景:上传待更新记录
- **当** 本地记录的 `sync_status` 为 `PENDING_UPDATE`
- **那么** 同步器必须把该记录的最新字段更新到对应的 Supabase 表
- **那么** 上传成功后必须把该记录本地状态切换为 `SYNCED` 并清空 `sync_error`

#### 场景:上传待删除记录
- **当** 本地记录的 `sync_status` 为 `PENDING_DELETE`
- **那么** 同步器必须把 `is_deleted = true` 的软删除状态上传到对应的 Supabase 表
- **那么** 同步器禁止在上传阶段直接物理删除云端记录

#### 场景:避免重复上传
- **当** 本地记录的 `sync_status` 为 `SYNCING`
- **那么** 其他同步任务必须跳过该记录
- **那么** 系统必须避免同一条数据被并发重复上传

### 需求:同步失败必须保留可重试状态
当任一记录同步失败时，系统必须保留失败原因和重试信息，使后续自动重试或用户手动重试可以继续处理。

#### 场景:上传失败
- **当** 同步器上传本地记录失败
- **那么** 系统必须把该记录本地 `sync_status` 设置为 `FAILED`
- **那么** 系统必须写入 `sync_error`
- **那么** 系统必须递增 `retry_count`

#### 场景:重试失败记录
- **当** 同步器选择重试 `FAILED` 记录
- **那么** 系统必须再次进入 `SYNCING`
- **那么** 重试成功后必须把该记录切换为 `SYNCED`、清空 `sync_error` 并重置 `retry_count`

### 需求:云端变更必须增量拉取到 Room
系统必须在登录后通过增量拉取把当前用户的 Supabase 数据写入本地 Room，并保持 UI 继续从 Room 派生。

#### 场景:应用启动后拉取云端数据
- **当** 用户已登录且应用启动
- **那么** 系统必须从 Supabase 拉取当前用户在上次拉取后变化的核心数据
- **那么** 系统必须在 Room transaction 中写入这些变化

#### 场景:拉取软删除记录
- **当** 云端返回 `is_deleted = true` 的记录
- **那么** 系统必须在本地保留该软删除标记
- **那么** 系统后续 UI 查询必须继续过滤已软删除数据

#### 场景:Realtime 通知触发拉取
- **当** 前台应用收到 Supabase Realtime 变更通知
- **那么** 系统必须触发对应数据范围的增量拉取
- **那么** 系统禁止只依赖 Realtime 通知本身作为最终数据源

#### 场景:设置页手动立即同步
- **当** 已登录用户在设置页触发立即同步
- **那么** 系统必须执行一轮上传本地待同步记录和拉取云端增量数据
- **那么** 系统禁止在主工作台页面暴露立即同步入口

### 需求:同步必须按关系依赖顺序处理
系统必须在上传和应用远端变化时尊重知识库表之间的依赖关系，避免关系记录先于主体记录导致引用缺失。

#### 场景:上传本地变更
- **当** 同步器同一轮需要上传多类记录
- **那么** 系统必须先上传知识树节点和标签
- **那么** 系统必须再上传摘录
- **那么** 系统必须最后上传摘录标签关系

#### 场景:关系记录引用暂缺
- **当** 拉取或上传摘录标签关系时依赖的摘录或标签尚不存在
- **那么** 系统必须保留该关系记录的待处理状态或失败状态
- **那么** 系统必须允许后续同步轮次继续重试

### 需求:多端冲突必须标记并按覆盖策略处理
当本地未同步变更与云端同一记录的远端变更发生冲突时，系统必须标记冲突，并在 MVP 阶段使用覆盖策略得到当前有效数据。

#### 场景:检测到冲突
- **当** 本地记录在 `last_sync_time` 后发生本地变更
- **且** Supabase 中同一 `id` 记录也在该时间后发生变化
- **那么** 系统必须把本地记录的 `sync_status` 设置为 `CONFLICT`

#### 场景:冲突覆盖
- **当** 系统处理冲突记录
- **那么** 系统必须比较本地与云端记录的 `update_time`
- **那么** 系统必须使用 `update_time` 较新的版本覆盖当前本地字段
- **那么** 当时间相同或无法可靠比较时，系统必须使用云端版本覆盖当前本地字段

#### 场景:冲突不提供人工合并
- **当** 第一版系统检测到冲突
- **那么** 系统禁止要求用户进行字段级手动合并
- **那么** 系统必须保留 `CONFLICT` 状态作为后续扩展冲突详情和人工解决的入口
