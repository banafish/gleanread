# 规范：fast-capture-widget
## 目的
GleanRead 快速捕获悬浮窗 (Fast Capture Widget) 作为整个系统的移动端前哨站，必须能以最低的阻断感接管原生系统的分享与文本处理意图，实现用户在摘录化阅读心流中的极慢网摘捕获与快速标记，并在来源字段不足时安全补齐上下文，确保知识采集过程顺滑且高效。
## 需求
### 需求:外部快摘必须支持系统文本处理入口
应用必须注册系统级文本处理入口，并以与分享入口一致的半屏快摘弹窗响应。

#### 场景:从系统文本处理菜单唤起快摘
- **当** 用户在第三方应用中选中文字并通过系统“处理文本”菜单选择 GleanRead
- **那么** 系统必须打开与分享入口一致的 `FastCaptureActivity` 半屏弹窗
- **那么** 系统必须将被选中的文本作为快摘正文预填到弹窗中

### 需求:外部快摘必须支持最近页面上下文回退补齐
当分享或文本处理入口缺失来源标题或 URL 时，系统必须按优先级尝试消费最近页面上下文缓存。

#### 场景:使用最近页面补齐来源
- **当** `ACTION_SEND` 或 `ACTION_PROCESS_TEXT` 入口未提供完整的来源标题或 URL
- **并且** 最近页面上下文缓存未过期且来源应用匹配当前入口
- **那么** 系统必须使用缓存中的标题或 URL 补齐快摘弹窗初始来源信息

#### 场景:最近页面上下文不可用时降级
- **当** 最近页面上下文缓存不存在、已过期或与当前入口来源应用不匹配
- **那么** 系统必须继续使用标准 `Intent` 解析结果与手动编辑降级路径
- **那么** 系统不得因为补齐失败阻塞快摘保存

### 需求:系统接管与极速浮层分享
应用必须注册系统级分享意图与文本处理意图，并通过半透明浮层响应，使用户可以快速摘录文本与附加个人想法；当标准 `Intent` 字段不足时，系统必须尝试使用最近页面上下文缓存补齐来源信息。

#### 场景:文本分享与极速采集（更新）
- **当** 用户在第三方应用中选中文本并通过系统分享或系统文本处理入口唤起 GleanRead
- **那么** 系统必须优先通过 `UrlExtractor` 和入口字段解析正文、URL 与标题
- **那么** 当入口字段不足时，系统必须尝试使用最近页面上下文缓存补齐来源标题和 URL
- **那么** 系统必须弹出一个不遮挡全文阅读的轻量浮窗，允许用户输入“想法”与多选“标签”后一键保存到本地知识库

### 需求：Intent 解析委托给 UrlExtractor
`FastCaptureActivity` 的 Intent 解析逻辑必须委托给 `UrlExtractor`，不再包含任何内联的 URL 提取判断代码。

#### 场景：URL 提取委托
- **当** `FastCaptureActivity.onCreate` 接收到 `ACTION_SEND` Intent 时
- **那么** URL 的提取必须通过调用 `UrlExtractor.extract(intent)` 完成

### 需求：双场景 URL 感知
系统必须能够区分并处理「整页分享」「选中文字分享」与「系统文本处理」三类入口的来源信息差异，并在标准 `Intent` 与最近页面上下文缓存之间按优先级补齐 URL 和标题。

#### 场景：整页分享时的 URL 自动填充
- **当** 用户使用浏览器的「分享整页」功能分享到 GleanRead 时
- **那么** 弹窗必须优先从标准 `Intent` 字段中提取并填充来源 URL
- **那么** 若 `Intent` 已提供完整来源信息，系统禁止再用缓存覆盖它

#### 场景：选中文字入口的来源补齐
- **当** 用户通过分享或文本处理入口只传入选中文字正文
- **那么** 系统必须先尝试从 `Intent` 文本、附加字段或 `clipData` 提取 URL
- **那么** 若标准提取仍然不足，系统必须继续尝试使用最近页面上下文缓存补齐标题和 URL

#### 场景：来源完全缺失时降级
- **当** 标准 `Intent` 解析与最近页面上下文缓存都无法提供有效 URL
- **那么** 弹窗必须展示可编辑的链接输入路径
- **那么** 用户不填写 URL 时仍然必须能够完成快摘保存

### 需求: 极速唤起
接收到意图后，必须不打开完整的 App 主界面，而是以半屏弹窗 (Transparent Activity/Dialog) 的形式弹出轻量级表单。如果在主界面内部唤起，则行为视内部路由而定；但作为外部 `ACTION_SEND` 响应时，必须保持原应用上下文可见。

#### 场景: 系统分享接收
- **当** 用户在其他应用（如浏览器）中选中文字并点击分享至 GleanRead
- **那么** 系统必须以底层应用为背景，弹出半屏的捕获表单，而不是打开全屏的 App 主界面。

#### 场景: 提交后关闭
- **当** 用户在弹出的半屏捕获表单中点击“确认保存”或背景遮罩取消时
- **那么** 弹窗必须通过结束自身（finish）来消失，使用户回到最初发起分享的应用界面。

### 需求: 沉浸式毛玻璃视觉 (Immersive Glassmorphism Context)
应用在分享唤起时，背景**必须**采用具有通透感且带模糊效果（Blur Effect）的材质，使摘录界面表现为源应用上方的「悬浮层」。

#### 场景: 分享唤起时的背景融合
- **当** 用户从任何第三方 App 分享文本到 GleanRead 时
- **那么** 弹窗底部必须呈现为圆角（至少 `28.dp`）且带背景模糊的毛玻璃效果。

### 需求: 现代化的书摘卡片化 (Modernized Excerpt Card)
摘录的原文内容**必须**以独立的「卡片」形式呈现在弹窗核心位置，而非简单的输入框。

#### 场景: 原文内容展示
- **当** 文本被成功提取入弹窗时
- **那么** 它必须以轻度阴影（Tonal Elevation）和优雅排版显示在核心卡片中。

### 需求: 胶囊标签与交互反馈 (Pill Tags & Interaction Feedback)
标签输入**必须**采用「胶囊」化的视觉表现，并在交互时提供丰富的反馈。

#### 场景: 快速打标
- **当** 用户点击预设标签或确认标签时
- **那么** 标签必须以带有呼吸感的动效状态出现在输入框附近。

### 需求: 阴影渲染一致性 (Shadow Rendering Consistency)
所有在 `FastCaptureActivity` 中使用的弹出层组件，其阴影必须在动画周期内严格遵守其几何形状。

#### 场景: 标签层弹出与收起
- **当** 用户点击左下角标签图标，且标签层正在通过 `ScaleIn` 动画显示时
- **那么** 其外围阴影必须从动画开始就具有圆角属性，严禁出现先显示方角阴影再变换为圆角的情况。

### 需求: 标签层动态交互限高与滚动 (Dynamic Tag Cloud Scrolling)
标签层组件必须具备自适应和限制高度的能力，以平衡显示信息量与输入区可视性。

#### 场景: 极多标签显示
- **当** 预设标签数量超过 8 个且在特定分辨率下超过两行高度时
- **那么** 标签显示区域的最大高度必须限制在约两行半左右（~130dp），并允许在其内部进行垂直滚动。
- **那么** 滚动时内容不应遮挡周围边框或标题。

#### 场景: 默认显示
- **当** 标签数量较少（如少于 5 个）且显示高度低于最大限制时
- **那么** 标签层高度应自适应收缩，保持紧凑。
