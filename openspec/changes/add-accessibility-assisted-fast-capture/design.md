## 上下文

当前 `FastCaptureActivity` 只消费系统分享 `Intent` 中的正文与 URL，并把它们直接传入快摘弹窗。这个设计在“整页分享”里还能工作，但在浏览器和微信公众号的“选中文字分享”场景下，`Intent` 往往只包含正文，不包含页面标题和链接；现有 `UrlExtractor` 已经覆盖了常见 `EXTRA_TEXT`、`EXTRA_SUBJECT`、`clipData` 组合，但无法解决宿主 App 根本不暴露这些字段的情况。

项目当前没有无障碍服务、没有 `ACTION_PROCESS_TEXT` 入口，也没有一条“来源标题”从快摘入口写入本地摘录的完整链路。与此同时，工作台摘录卡片已经具备 `sourceTitle ?: url` 的展示位，因此这次设计既要补一个新的页面上下文采集通道，也要把已有 UI 和持久化链路真正串起来。

约束如下：
- Android 端必须保持本地优先，不得引入抓网页或元数据解析的网络请求。
- 无障碍能力必须是显式开启、可关闭、可降级的辅助模式，不能变成快摘主路径的硬依赖。
- 浏览器与微信 WebView 暴露的节点结构不稳定，方案必须接受“拿不到 URL/标题”的现实，并保留手动输入降级。

## 目标 / 非目标

**目标：**
- 为外部快摘新增一条独立于分享 `Intent` 的“最近页面上下文缓存”能力，短时保存最近可识别的标题、URL、来源应用与采集时间。
- 为快摘新增 `ACTION_PROCESS_TEXT` 入口，覆盖系统文本处理入口。
- 在 `Intent` 信息不足时，使用最近页面上下文缓存补齐 URL 和标题，并以明确的优先级规则合并。
- 打通 `sourceTitle` 与 `url` 的保存链路，使补齐后的来源信息能写入本地摘录并在工作台展示。
- 为无障碍增强模式补充非阻塞的权限引导和状态提示。

**非目标：**
- 不尝试通过网络抓取网页元数据或反查短链。
- 不承诺所有浏览器和所有微信页面都能稳定暴露 URL；无法获取时仍以手动输入为最终降级路径。
- 不把无障碍缓存扩展成长期浏览历史，不做跨天持久化与搜索能力。
- 不改造主工作台 FAB 打开的内部快速记录面板行为；本次聚焦外部分享与文本处理入口。

## 决策

### 决策一：引入独立的 `PageContextSnapshot` 短时缓存，而不是继续扩展 `UrlExtractor`

`UrlExtractor` 保持“只从 `Intent` 提取 URL”的职责，不承担无障碍或缓存回退逻辑。新增一个独立的 `PageContextSnapshot` 模型与 `PageContextStore`，用于保存最近一次识别到的：
- `sourcePackage`
- `sourceTitle`
- `sourceUrl`
- `capturedAt`
- `captureSource`（如 `accessibility`）
- `confidence`

快摘入口在组装初始数据时通过 `CaptureSeedResolver` 合并两类来源：
1. `Intent` 中的明确字段
2. 短时缓存中的最近页面上下文

优先级固定为：
1. `Intent` 中的显式正文 / URL / 标题
2. 同一来源应用、未过期的页面上下文缓存
3. 用户在弹窗中手动编辑的链接

选择这个方案而不是继续扩展 `UrlExtractor`，是为了避免把“分享字段解析”和“运行时页面观察”耦合进同一个工具类，也便于分别测试 `Intent` 解析与上下文合并规则。

考虑过的替代方案：
- 继续把缓存回退堆进 `UrlExtractor`：职责混乱，且会让 `UrlExtractor` 偏离“纯 `Intent` 提取器”的定义。
- 在 `FastCaptureActivity` 里直接写合并逻辑：能做，但会让 Activity 同时承担路由、解析、状态合并和 UI 初始化，后续难以测试。

### 决策二：页面上下文缓存只保存单条最近快照，并设置严格 TTL

页面上下文缓存只保留“最近一条”快照，并为消费设置严格有效期。当前先固定为 1 分钟内可读，命中后可按需清空或更新时间戳。消费时必须额外校验：
- `sourcePackage` 与当前分享 / 处理文本来源一致，或至少属于允许的宿主列表
- 缓存未过期
- 缓存中的标题 / URL 通过基本合法性校验

之所以不用 Room 保存多条历史，是因为这个缓存只服务于“从原页面跳入快摘”的短时补齐，不属于知识库的正式领域数据。单条短时缓存也更符合最小权限和最小暴露原则。

考虑过的替代方案：
- 使用 Room 持久化历史页面上下文：能力过重，且会把临时浏览上下文混入正式知识数据。
- 完全只用内存缓存：进程被回收或服务/Activity 分离时命中率不稳定。

### 决策三：无障碍服务是可选增强通道，不自动替代标准分享入口

新增一个专用 `AccessibilityService`，仅在用户显式开启后生效。该服务监听前台窗口变化与内容变化，针对浏览器和微信等已知宿主尝试提取页面标题、URL 和来源应用，并写入 `PageContextStore`。

服务只做“观察并缓存”，不直接弹出快摘弹窗，也不主动拦截分享行为。这样可以把风险控制在最小范围内：即便服务不可用，标准 `ACTION_SEND` 与手动 URL 输入依旧可用。

微信公众号页面的首版识别策略只支持高置信度节点，例如标题栏、顶部工具栏和明显的地址展示区，禁止一开始就把正文周边的低置信度节点纳入来源识别。规则先按“宿主类型”维护，例如 `wechat-mp-browser`、`chrome-tab` 这类粒度；只有在后续确认某个宿主的不同版本存在稳定且不可忽略的结构差异时，才允许再细分到版本级规则。

考虑过的替代方案：
- 让无障碍服务直接触发快摘：时序和误触发风险高，且更难解释给用户。
- 不做无障碍，只加 `ACTION_PROCESS_TEXT`：可以覆盖“直接处理选中文本”，但依旧无法补齐标题和链接。
- 首版就覆盖更多正文周边节点或直接按微信版本维护规则：命中率可能更高，但误识别和维护成本都明显更大，不适合作为首发策略。

### 决策四：新增 `ACTION_PROCESS_TEXT` 入口，但与分享入口复用同一套组装流程

Manifest 中新增 `ACTION_PROCESS_TEXT` 入口，选中文本后直接进入 `FastCaptureActivity`。`ACTION_PROCESS_TEXT` 与 `ACTION_SEND` 最终都走同一个 `CaptureSeedResolver`，区别只在于正文来源字段不同。

这样做可以避免为两个入口分别维护一套快摘初始化逻辑，也能保证后续测试集中在“输入源解析”和“来源补齐合并”两层。

考虑过的替代方案：
- 单独新增一个 `ProcessTextActivity`：入口隔离清晰，但会复制快摘初始化与 UI 组装逻辑。

### 决策五：快摘正式持久化只新增来源标题贯通，不把缓存元数据写入摘录表

正式保存到 `ExcerptEntity` 的仍然只有业务上长期有价值的字段：
- `content`
- `url`
- `sourceTitle`
- `userThought`
- 标签与归档关系

缓存专用元数据如 `confidence`、`captureSource`、`capturedAt` 保留在临时缓存层，不进入正式摘录表。这样可以避免本地知识库承担调试/辅助性质字段，也无需扩大工作台展示和同步语义。

考虑过的替代方案：
- 把 `captureSource`、`confidence` 一并写入摘录表：便于排查，但会扩大数据模型和 UI/同步语义，本期收益有限。

### 决策六：权限引导优先放在快摘弹窗的非阻塞提示位，并预留主工作台帮助入口

当快摘发现：
- 标题或 URL 缺失
- 当前设备尚未开启无障碍增强
- 当前来源应用属于支持增强识别的宿主

弹窗展示一条非阻塞提示，说明可以开启“辅助识别模式”以尝试自动补齐来源，并提供跳转系统无障碍设置的入口。该提示不能阻塞当前保存流程，用户可以直接忽略并手动补链。后续还要在主工作台补充一个帮助入口，用于解释该能力的用途、限制和开启方式。

考虑过的替代方案：
- 新增独立设置页：长期可能需要，但当前项目没有现成设置模块，会显著扩大本次范围。
- 首次启动强制引导开启无障碍：侵入性过高，不符合“可选增强”的定位。

## 风险 / 权衡

- [风险] 微信 WebView 或某些浏览器不暴露 URL 节点，导致无障碍增强也补不出链接。 → 缓解措施：把 URL 视为“尽力而为”的增强结果，始终保留手动输入降级。
- [风险] 页面上下文缓存误命中到上一页，导致来源链接串页。 → 缓解措施：增加 TTL、来源应用校验、URL 合法性校验，并要求只有在 `Intent` 缺失字段时才消费缓存。
- [风险] 无障碍权限门槛高，用户不愿开启。 → 缓解措施：保持标准分享入口完整可用，把无障碍定位为可选增强，不把主流程绑定到该权限。
- [风险] 服务高频监听窗口变化可能带来噪音与误采集。 → 缓解措施：只针对白名单宿主做解析，并对相同标题/URL 的重复快照做去重。
- [风险] `ACTION_PROCESS_TEXT` 与 `ACTION_SEND` 的正文字段格式不同，若入口解析分散容易产生回归。 → 缓解措施：所有入口统一收敛到 `CaptureSeedResolver` 与测试用例。

## 迁移计划

1. 先增加 `CaptureSeedResolver`、`PageContextStore` 与 `ACTION_PROCESS_TEXT` 入口，保持现有 `ACTION_SEND` 行为不变。
2. 打通 `sourceTitle` 从快摘入口到 `WorkspaceRepository.saveQuickExcerpt` 的保存链路。
3. 引入无障碍服务与快摘提示，但默认关闭，不影响现有用户。
4. 在快摘入口上灰度验证：无障碍关闭时分享与保存流程必须与现状兼容；无障碍开启时仅在字段缺失时启用缓存回退。
5. 如果出现串页或误识别，可通过关闭服务声明与忽略缓存回退逻辑快速回滚，不影响分享主路径。
