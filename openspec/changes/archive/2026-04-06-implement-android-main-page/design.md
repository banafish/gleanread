## 上下文

当前 Android 客户端只有两个入口：`MainActivity` 仍是占位文本页，`FastCaptureActivity` 只覆盖外部分享场景。用户补充要求后，本次变更范围已经明确：

- **实现重点是安卓本地端**：首版不依赖现有后端接口；摘录保存、AI 提炼流程、知识树整理和图谱展示全部以本地数据为准，后续再补云同步。
- **UI 约束非常强**：Compose 页面必须严格遵循 `doc/ui.tsx` 的页面结构、导航关系、视觉层次和关键交互，而不是只做近似实现。
- **数据约束明确**：本地存储必须遵循给定的 Room 四表结构（`knowledge_tree_node`、`tags`、`excerpts`、`excerpt_tags`），并保留 `sync_status` / `is_deleted` 等离线优先字段。
- **首版体验约束补充**：数据库为空时需要提供示例数据与空状态引导；主界面 FAB 只写本地库并标记待同步；节点详情页的“局部图谱”不是占位按钮，而是需要完整可交互视图。

受影响的核心区域包括：
- `glean-read-android/app/src/main/java/com/gleanread/android/MainActivity.kt`
- Android 端新增的 Room 实体、DAO、Database、Repository、ViewModel 与 Compose 页面
- `doc/ui.tsx` 对应的页面拆分、交互还原与样式 token 映射

## 目标 / 非目标

**目标：**
- 按 `doc/ui.tsx` 严格实现 Android 主工作台，覆盖摘录流、知识树、标签、AI 提炼页、节点详情页、FAB 快速记录与底部导航。
- 用 Room 建立本地优先的数据闭环，让主界面所有核心读写都只依赖本地库，并通过 `sync_status` 为后续云同步预留入口。
- 提供内置示例数据和空状态引导，保证新安装或清空数据库后仍能完整展示产品结构与交互。
- 提供不依赖后端的本地 AI 大纲工作流，覆盖多选摘录、生成大纲草稿、选择/新建节点、写回知识树。
- 在节点详情页完成可交互的局部关系图，支持查看节点、摘录和引用关系的连接结构。
- 在摘录想法和节点大纲编辑中提供 `[[` 联想、插入和渲染能力。

**非目标：**
- 本次不接入现有服务端接口，也不实现完整的云同步、账号体系和冲突解决。
- 本次不改造服务端 API 或数据库结构。
- 本次不引入重量级富文本框架、图数据库或额外的 Room 关系索引表。
- 本次不要求真正调用远端大模型；首版 AI 提炼能力只需要在本地端完成可用流程与可替换边界。

## 决策

### 决策 1：采用单 Activity + Navigation Compose，并以 `doc/ui.tsx` 作为唯一 UI 蓝本
- **选择**：保留 `MainActivity` 作为应用内主入口，在其中使用 `NavHost` 承载 `feed`、`tree`、`tags`、`aiSummary`、`nodeDetail` 等页面；顶层 Scaffold 控制底部导航与 FAB 显隐。
- **原因**：`doc/ui.tsx` 已经给出完整的页面流和层级关系，Navigation Compose 能稳定表达页面切换、返回栈和参数传递，并让 Compose 页面按设计稿拆分为明确的 route。
- **实现约束**：页面结构、按钮位置、列表分区、Bottom Sheet 行为、多选操作栏、节点选择器和详情页信息顺序都以 `doc/ui.tsx` 为准，避免“功能对了但布局不同”的偏差。
- **备选方案**：
  - 手写单状态机切页面：短期可行，但随着详情跳转、局部图谱和多层返回出现，状态会变得脆弱。
  - 多 Activity：难以共享底部导航、FAB 和跨页面本地状态，不适合这个工作台场景。

### 决策 2：主工作台采用 Room + Repository + ViewModel 的本地优先架构
- **选择**：新增 Room 实体、DAO、Database 和 Repository；所有主界面页面都通过 ViewModel 订阅 Room `Flow`，把本地数据库作为唯一读模型。
- **原因**：用户已明确首版主界面只处理本地存储，后续再加云同步；直接把数据存在 Compose state 或继续依赖当前后端接口都不符合范围。
- **实现约束**：
  - 用户创建/修改/删除的数据写入本地库时，统一维护 `sync_status`。
  - FAB 新建摘录默认写入 Inbox（`tree_node_id = null`），并标记 `PENDING_CREATE`。
  - 页面展示只从本地查询，不等待网络结果。
- **备选方案**：
  - 网络优先：与“首版不调用后端接口”直接冲突。
  - 仅用内存假数据：无法形成真正的本地工作台和后续同步边界。

### 决策 3：AI 提炼流程通过本地可替换的 OutlineGenerator 完成，不调用后端接口
- **选择**：在 Android 端定义 `OutlineGenerator` 抽象，并先提供本地实现（如基于已选摘录、标签和标题规则生成结构化大纲草稿）；多选摘录进入 AI 提炼页时只调用本地生成器。
- **原因**：用户明确要求本次不依赖现有后端接口，但仍需要完整的 AI 提炼交互链路；把生成逻辑抽象在本地接口后，未来可平滑替换为真实远端 AI / 同步实现。
- **实现约束**：
  - 生成结果必须是可编辑文本，并能继续挂载到已有节点或新建节点。
  - 挂载动作只改本地库：更新节点、写入摘录的 `tree_node_id`，必要时新建知识树节点。
- **备选方案**：
  - 直接调用当前服务端 AI 大纲接口：超出本次范围。
  - 完全写死 mock 文本：无法根据用户已选摘录变化，不利于后续替换。

### 决策 4：空数据库时先展示引导，再支持一键导入示例数据
- **选择**：当 Room 中不存在用户数据时，先展示空状态引导页/空列表引导，并提供“加载示例数据”和“开始记录”两个入口；示例数据通过独立 Seeder 写入同一套 Room 表结构。
- **原因**：用户要求同时满足“内置示例数据”和“空状态引导”。先给引导，再按需导入示例数据，可以避免首次启动就把演示数据混入真实数据，同时仍能快速展示完整 UI。
- **实现约束**：
  - 示例数据结构要与 `doc/ui.tsx` 展示内容一致，确保 UI 演示真实贴合设计稿。
  - 示例数据通过显式动作写入，便于后续清空或替换。
- **备选方案**：
  - 首次启动自动灌入示例数据：演示效果直接，但会污染真实本地库。
  - 只显示空状态不提供示例数据：无法满足用户补充要求。

### 决策 5：`[[` 双向链接以结构化文本保存，展示时解析为可点击高亮内容
- **选择**：在 `user_thought` 和 `outline_markdown` 中保存 `[[id:<targetId>|<title>]]` 形式的文本；编辑时使用 `TextFieldValue` 监听 `[[` 触发联想面板，展示时解析为 `AnnotatedString` 并支持点击跳转。
- **原因**：这能在不引入富文本框架的前提下同时满足“联想插入”“持久化带 ID 的引用”和“详情页高亮跳转”三类需求。
- **备选方案**：
  - 只保存 `[[title]]`：标题重名或改名时无法稳定定位目标。
  - 额外建立链接关系表：会增加模型复杂度，超出当前范围。

### 决策 6：节点详情页的局部图谱采用 Compose Canvas + 手势交互实现
- **选择**：在节点详情页新增局部图谱视图，使用 Compose Canvas 绘制节点与边，配合 `pointerInput` / `transformable` 支持拖拽、缩放、点击聚焦；图谱节点来自本地知识树节点、关联摘录和 `[[` 引用解析结果。
- **原因**：用户已经明确要求首版同时完成可交互图谱。自绘可以避免引入额外图形库，也更容易控制视觉风格与点击行为。
- **实现约束**：
  - 中心节点固定为当前详情节点。
  - 周边节点至少包含：被当前节点引用的节点、引用当前节点的节点、挂载在当前节点下的摘录。
  - 点击图谱中的节点或摘录要能联动跳转或打开详情。
- **备选方案**：
  - 仅放占位按钮或静态截图：不满足要求。
  - 引入第三方图谱库：依赖更重、样式可控性更差。

### 决策 7：主界面 FAB 复用统一摘录草稿模型，但首版只做本地写入
- **选择**：Bottom Sheet 内的正文、URL、标签、归档位置都通过统一的本地草稿/保存 use case 写入 Room，不发任何网络请求。
- **原因**：用户已明确要求 FAB 保存只写本地库并标记待同步；统一保存路径还能让 Inbox、标签热度、节点挂载状态即时刷新。
- **实现约束**：
  - 默认归档位置为 Inbox。
  - 标签热度在本地保存时即时累加。
  - 关闭面板与保存成功反馈都依赖本地事务结果。
- **备选方案**：
  - 继续沿用现有网络保存逻辑：与当前范围冲突。
  - 分散写入多个页面状态：难以维护一致性。

## 风险 / 权衡

- **[严格对齐 `doc/ui.tsx` 会让 Compose 代码量增大]** → 通过提取共享卡片、底部导航、FAB Sheet、节点行和标签胶囊组件，减少重复实现，同时保证布局一致。
- **[本地 OutlineGenerator 不是“真正 AI”]** → 先以可替换接口完成完整流程，保证页面与数据链路稳定；后续接入远端模型时只替换实现层。
- **[示例数据与真实数据可能混淆]** → 通过空状态显式触发导入示例数据，并提供清空入口或可识别的示例标识，避免默认混入。
- **[局部图谱自绘需要处理布局与手势性能]** → 首版采用有限节点数量和缓存后的布局结果，必要时限制默认展示深度，优先保证交互稳定。
- **[没有后端同步时，本地状态可能长期停留在待同步]** → repository 始终维护 `sync_status` 字段，并把未来同步入口隔离在数据层，不影响当前 UI 读写。

## 迁移计划

1. 在 `app/build.gradle.kts` 中加入 Room、Lifecycle、Navigation Compose 等依赖，并配置 Room 编译器。
2. 新增 Room 数据层：实体、DAO、Database、Repository、查询模型与示例数据 Seeder。
3. 将 `MainActivity` 替换为 Workspace 根页面，先接通摘录流、知识树、标签页、底部导航与 FAB。
4. 接入本地 AI 提炼页、节点详情页、`[[` 联想编辑器和局部图谱视图，并把结果全部写回 Room。
5. 打通空状态引导与示例数据导入，让首次进入应用时既可直接开始记录，也可加载演示工作台。
6. 后续如需接入云同步或远端 AI，只替换 Repository/OutlineGenerator 的实现，不改动页面路由和 Room 读模型。

## 开放问题

- 暂无。本次范围已经明确：本地优先、严格遵循 `doc/ui.tsx`、FAB 只写本地库、空库提供引导与示例数据、节点详情页实现可交互局部图谱。
