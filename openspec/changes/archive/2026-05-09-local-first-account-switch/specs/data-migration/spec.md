## 新增需求

### 需求:访客数据向正式账号迁移
当用户从访客状态转化为登录状态时，系统必须询问并允许用户将本地生成的临时数据迁移至新账号数据库中。

#### 场景:访客有数据并选择合并
- **当** 用户登录成功且本地 `guest.db` 存在业务数据，且用户同意合并数据时
- **那么** 系统必须将 `guest.db` 中的记录写入新账号对应的 `local_data_{user_id}.db` 中，并启动后台同步至云端

#### 场景:访客无数据或拒绝合并
- **当** 本地 `guest.db` 无有效业务数据，或用户拒绝合并时
- **那么** 系统禁止合并数据，并继续执行常规登录流程

#### 场景:合并后的访客库清理
- **当** 数据合并成功后，登录流程完成时
- **那么** 系统必须清空并关闭 `guest.db`，确保再次进入访客态时处于干净状态

#### 场景:拒绝合并后保留访客库
- **当** 用户拒绝将 `guest.db` 数据合并到当前账号后，登录流程完成时
- **那么** 系统必须保留 `guest.db` 中的原有访客数据
- **那么** 用户再次进入访客态时必须能够继续查看和编辑这些访客数据
