## 上下文

应用采用 Local-First 架构，将本地数据库作为单一数据源 (SSOT) 直接支持 UI 读写。目前面临的核心挑战是如何在 Local-First 模式下优雅地处理不同认证状态（访客、已登录账号）之间的数据隔离、迁移、多账号频繁切换的体验（0 延迟），以及离线状态下的未同步数据保护。

## 目标 / 非目标

**目标：**
- 实现无缝的访客离线体验（通过独立的 `guest.db`）。
- 实现访客数据到正式账号的平滑合并迁移。
- 实现安全的账号登出流程，防止离线未同步数据遗失。
- 实现单设备多账号安全隔离和零延迟（秒开）切换。
- 提供本地存储（废弃 `.db` 文件）的主动与被动清理机制。

**非目标：**
- 复杂的端到端 (P2P) 数据同步机制（统一通过 Supabase 同步）。
- 自定义的服务端数据冲突解决策略（依赖当前的 Sync 引擎解决）。

## 决策

1. **按用户分库设计 (File-per-user DB Isolation)**
   - **决策**：为访客创建 `guest.db`，为每个登录用户创建独立的 `local_data_{user_id}.db`。
   - **理由**：相比于“单库 + `user_id` 列”的方案，独立文件隔离提供了物理级别的数据安全，避免了复杂查询和错乱；同时在登出和账号切换时只需切换数据库连接实例，能实现真正的“零延迟秒开”。清理历史账号数据也变得简单（直接删除文件）。
   - **替代方案**：单库多租户设计，开发简单但查询开销大，且容易导致不同用户数据意外混淆，账号切换时需清理内存状态或执行复杂的过滤。

2. **退出时不删除本地 DB 文件**
   - **决策**：用户注销时仅断开连接并断开同步，保留 `local_data_{user_id}.db` 文件。
   - **理由**：下次该账号重新登录时，可瞬间从本地 DB 读取历史数据并直接渲染 UI，同时在后台静默发起增量同步。
   - **权衡**：会增加本地存储空间占用。
   - **缓解**：通过增加“历史缓存数据库管理”机制，自动清理 30 天未活动的账号数据库。

3. **登出前的数据队列拦截 (Sync Queue Guard)**
   - **决策**：用户触发“退出登录”时，系统强制检查当前数据库的同步队列。若有未 Push 完成的数据，弹窗警告用户。
   - **理由**：在离线或弱网下修改了数据，若直接登出，这些数据将仅存在于本机的独立 DB 文件中。如果不加警告，用户在其他设备上将无法看到这些修改。
   - **替代方案**：强制后台同步直至完成。但在持续无网环境下无法实现，需交由用户自行判断。

4. **拒绝合并时保留访客库**
   - **决策**：用户登录后选择不合并访客数据时，系统保留 `guest.db` 中的原有数据，仅让当前账号继续使用自己的 `local_data_{user_id}.db`。
   - **理由**：`KEEP_LOCAL` 表示这些内容仍归属访客工作区，而不是被当前账号接管或丢弃；用户之后回到访客态时应能继续访问这些本地内容。
   - **权衡**：后续登录其他账号时仍可能再次看到合并选择，需要 UI 文案明确这是访客工作区数据。

## 风险 / 权衡

- [空间占用过大] → 多个账号登录会导致生成多个数据库文件。缓解：引入主动垃圾回收策略，在启动时检查并删除超期未活跃的 `local_data_*.db` 文件。
- [访客数据合并冲突] → 访客操作的数据在合并至一个已有大量云端数据的正式账号时，ID 可能冲突或重复。缓解：使用 UUID 作为主键，确保访客态生成的数据合并时天然不冲突；关联关系在合并时进行统一 re-parent。
