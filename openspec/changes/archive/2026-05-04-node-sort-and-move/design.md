## 上下文

当前知识树节点的排列完全依赖 `create_time`——DAO 查询使用 `ORDER BY create_time ASC`，`NodeProjectionFactory` 按插入顺序构建 `childNodeIds`。用户无法自定义同级节点顺序。

现有的移动功能（`MoveNodeBottomSheet`）只修改节点的 `parentId`，未涉及目标位置选择或排序值更新。移动后节点默认出现在目标父节点子列表的末尾（按 `create_time`），用户无法控制插入位置。

本次变更需要在数据层引入稀疏排序字段，在 Repository 层实现排序值计算逻辑，在 UI 层提供长按整张卡片触发的同级拖拽排序交互（拖拽过程中其他节点视觉上相应移动以显示插入位置）。

知识树页面结构说明：首页最多展示根节点 + 2 层预览（共 3 层），分支页最多展示当前节点 + 3 层预览。超出可见深度的节点需要进入下一级分支页浏览。拖拽排序仅支持同级排序移动，跨层级移动由现有的 `MoveNodeBottomSheet` 承担。

## 目标 / 非目标

**目标：**
- 为知识树节点引入 `sort_order` 稀疏排序字段，同级节点按此字段排列
- 新增节点时自动分配排序值（追加到同级末尾）
- 长按整张节点卡片进入拖拽模式，仅支持同级排序移动（不支持跨层级移动，不支持单独拖拽卡片内子节点）
- 拖拽过程中其他节点根据被拖拽节点的位置视觉上相应移动（被挤上来或挤下去），以此直观显示和计算插入位置
- 拖拽结束后自动计算新的排序值
- 现有的 `MoveNodeBottomSheet`（菜单中的"移动"操作）保留不变，用于跨层级移动

**非目标：**
- 不实现跨页面拖拽（只在当前屏幕可见节点范围内拖拽）
- 不实现跨层级拖拽移动（跨层级移动由 `MoveNodeBottomSheet` 承担）
- 不实现单独拖拽卡片内部的子节点（只能拖拽整张卡片）
- 不实现批量排序/批量移动
- 本地数据库迁移不计算已有节点的初始排序值——初始值由 Supabase 脚本计算，App 通过同步获取

## 决策

### 决策 1：排序字段类型与稀疏策略

**选择**：使用 `Long` 类型的 `sort_order` 字段，初始间隔 `SORT_ORDER_GAP = 65536`。

**理由**：
- `Long` 范围（+/-9.2x10^18）提供了极大的排序空间，即使频繁在中间插入也不会很快耗尽
- 间隔 65536（2^16）在正常使用中可以连续在同一位置中间插入 16 次才会触发局部重排
- 相比 `Double`/`Float`，整数排序更精确且数据库索引效率更高
- 相比 `String`（如字典序排序键），实现和调试更直观

**替代方案**：
- `Int`：范围有限（+/-2x10^9），频繁操作后更容易溢出需要全局重排
- `Double`：浮点精度问题可能在极端情况下导致排序异常
- 分数排序（Fractional Indexing）：实现更复杂，在本项目当前规模下收益不明显

### 决策 2：排序值计算与局部重排策略

**选择**：三段式策略。

1. **追加到末尾**（创建节点、移动到末尾）：`lastSiblingSortOrder + SORT_ORDER_GAP`
2. **插入到两个节点之间**：`(prevSortOrder + nextSortOrder) / 2`
3. **局部重排**：当计算出的新值与相邻值差距 <= 1 时，对当前父节点下所有兄弟节点重新分配排序值（间隔恢复为 `SORT_ORDER_GAP`），在同一数据库事务中完成

**理由**：
- 正常操作只更新 1 行（被移动/创建的节点），性能最优
- 局部重排只在极端情况触发，且仅影响同一父节点下的兄弟节点，不影响其他子树
- 事务保证重排过程中不会出现中间状态

### 决策 3：拖拽排序交互设计

**选择**：长按整张节点卡片进入拖拽模式，仅支持同级排序移动，不支持跨层级移动和单独拖拽卡片内子节点。

**方案**：
- 长按整张节点卡片（不可单独拖拽卡片内部的子节点）-> 节点进入拖拽模式（视觉上抬起、添加阴影/缩放反馈）
- 拖拽仅支持同级排序移动，不支持跨层级移动（跨层级移动由 `MoveNodeBottomSheet` 承担）
- 拖拽过程中，被拖拽节点经过的位置，其他节点视觉上相应移动——被拖拽节点往下拖时下方节点往下挤，往上拖时上方节点往下让（类似其他卡片被挤上来或挤下去的效果）
- 通过其他节点的视觉位移来确定插入位置：当其他节点发生位移时，说明被拖拽节点已经移动到了该位置对应的两张卡片之间
- 拖拽结束后，根据视觉位移确定的插入位置计算前后兄弟节点的 `sort_order` 中间值
- 知识树页面可见深度有限（首页 3 层、分支页 3 层），拖拽范围自然受限于当前屏幕

**与现有移动功能的关系**：
- 现有的 `MoveNodeBottomSheet`（通过菜单中的"移动"操作触发）保留不变
- `MoveNodeBottomSheet` 定位为**跨层级移动**工具——用户可以浏览整个知识树选择目标父节点
- 拖拽排序定位为**同级快速调整**——直观、高效、适合微调同级节点顺序
- 两种交互互补：拖拽用于同级位置调整，BottomSheet 用于跨层级移动

**替代方案**：
- 支持跨层级拖拽：交互复杂度高，不同层级的卡片尺寸和布局差异大，跨层级拖拽容易误操作
- drop indicator 插入线：不如节点位移直观，用户需要理解线条位置对应哪个插入点
- 上下移动按钮：操作效率低，不适合连续调整

### 决策 4：长按触发与拖拽实现方式

**选择**：使用 Compose 的 `detectDragGesturesAfterLongPress` 手势检测实现拖拽，仅作用于整张卡片。

**方案**：
- 整张节点卡片通过 `Modifier.pointerInput` + `detectDragGesturesAfterLongPress` 检测长按后拖拽（不可单独拖拽卡片内部的子节点）
- 拖拽开始时触发 HapticFeedback，节点视觉状态变为"拖拽中"（elevation 提升、半透明化原位置占位）
- 拖拽过程中，其他同级节点根据被拖拽节点的 Y 轴位置发生视觉位移——被拖拽节点经过时，其他节点被"挤"开（向下或向上移动），通过位移量来计算被拖拽节点被移动到了哪两个节点之间
- 拖拽结束时执行排序值更新
- 拖拽仅限同级排序，不涉及 `parentId` 更新
- Route 层维护拖拽状态：`draggedNodeId: String?`、`dragOffsetY: Float`、`displacedNodeIds: List<String>` 等

**理由**：
- `detectDragGesturesAfterLongPress` 是 Compose 官方推荐的长按拖拽手势 API
- 现有 `RootNodeCard`、`PreviewNodeItem`、`BranchNodeItem` 可统一添加拖拽手势修饰符
- 不与现有的点击（进入分支/详情）、展开控件冲突——长按 + 拖拽与短点击互不干扰

### 决策 5：数据库迁移策略

**选择**：本地 `ALTER TABLE ADD COLUMN` 仅加字段赋默认值 0；已有节点的初始排序值由 Supabase 脚本计算，App 通过主动同步获取。

**方案**：
1. **本地迁移（MIGRATION_2_3）**：`ALTER TABLE knowledge_tree_node ADD COLUMN sort_order INTEGER NOT NULL DEFAULT 0`——仅加列，所有已有节点 `sort_order = 0`
2. **Supabase 端脚本**：在 Supabase 的 `knowledge_tree_node` 表新增 `sort_order` 列后，执行一次性脚本，按 `parent_id` 分组、`create_time ASC` 排序，为每个节点分配 `rowNumber * SORT_ORDER_GAP` 的初始值
3. **App 同步**：App 在下一次主动同步时拉取带有正确 `sort_order` 的节点数据，覆盖本地的默认 0 值
4. **新增节点**：App 端新增节点时仍会本地计算 `sort_order`（追加到末尾），不受迁移影响

**理由**：
- 本地迁移极其轻量，`ALTER TABLE ADD COLUMN` 无需重建表
- 避免在 SQLite 中写复杂的窗口函数初始化逻辑（SQLite 3.25+ 才支持窗口函数，部分旧设备可能不支持）
- 排序值的权威来源放在 Supabase 端，确保多端一致
- 数据库版本从 2 升到 3
- 在同步完成前，所有节点 `sort_order = 0`，排序退回 `create_time` 兜底

### 决策 6：Sync 层处理

**选择**：`sort_order` 字段加入 `RemoteKnowledgeTreeNode` 的序列化模型，同步时透传。

**理由**：
- `sort_order` 是节点的业务属性，需要多端一致
- 远端 Supabase 表需要新增对应列（此为后端变更，不在本次 Android 端实现范围内，但模型需要先支持）
- `toRemote()` 和 `toEntity()` 映射函数需要同步更新

## 风险 / 权衡

| 风险 | 缓解措施 |
|------|----------|
| 局部重排在大量兄弟节点场景下可能短暂卡顿 | 事务内批量 UPDATE，且正常知识树单层节点数不会很多（<100），影响可忽略 |
| 旧版本 App 不识别 `sort_order` 列 | Room 的 `ALTER TABLE ADD COLUMN` 迁移是向前兼容的；旧版本不读取该列，只会退回 `create_time` 排序 |
| 远端 Supabase 未同步新增列前，同步可能出现字段丢失 | `RemoteKnowledgeTreeNode.sortOrder` 设为可空并给默认值 0，确保序列化/反序列化兼容 |
| 同步前所有节点 `sort_order = 0`，排序可能混乱 | DAO 查询使用 `ORDER BY sort_order ASC, create_time ASC` 双重排序，`sort_order` 全为 0 时自动退回 `create_time` 顺序 |
| 长按拖拽可能与现有长按行为冲突 | 当前节点行无 `onLongClick` 处理，不存在冲突；`detectDragGesturesAfterLongPress` 与短点击互不干扰；拖拽仅作用于整张卡片，不与卡片内子节点的交互冲突 |
| 拖拽仅限同级，用户可能期望跨层级拖拽 | 跨层级移动由 `MoveNodeBottomSheet` 承担，拖拽定位为同级快速调整，功能互补 |
| 拖拽过程中节点展开/收起状态变化导致列表错乱 | 拖拽模式下禁用节点展开/收起操作，拖拽结束后恢复 |
| 稀疏排序值在极端操作下耗尽间隔 | 局部重排机制自动恢复间隔；`Long` 范围足够大，不会出现全局耗尽 |
