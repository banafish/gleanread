# 规范：web-knowledge-tree-graph

## 目的
定义 GleanRead Web 端中间知识树画布的布局、节点状态、键盘交互和视图控制，确保用户能够在横向知识树上快速建树、浏览和定位节点。

## 需求

### 需求:知识树画布必须以左到右的有向树形式渲染
系统必须将 `knowledge_tree_node` 渲染为从左到右展开的知识树画布，并以一个仅用于展示的虚拟根节点承载 `parent_id IS NULL` 的一级节点。画布中的连线必须保持平滑的树状连接关系，且同级节点必须按 `sort_order` 升序排列；当 `sort_order` 相同时，系统必须使用稳定的次级顺序兜底，避免节点在刷新后随机换位。

#### 场景:展示一级节点
- **当** 用户首次进入知识树画布
- **那么** 系统必须显示虚拟根节点以及当前可见的一级节点
- **那么** 系统不得一次性展开所有层级节点

#### 场景:刷新后保持节点顺序稳定
- **当** 用户重新进入工作台且节点数据未变化
- **那么** 系统必须按相同顺序重建知识树画布
- **那么** 同级节点不得因为重新渲染而产生顺序抖动

### 需求:知识树画布必须支持局部展开状态的本地记忆
系统必须只在本地保存知识树节点的展开与折叠状态，并在用户重新进入工作台时恢复上次的展开视图。初始加载时系统只需要展开第一层节点，其余层级必须保持折叠，除非用户显式展开对应节点。

#### 场景:首次加载只展开第一层
- **当** 用户第一次打开知识树画布
- **那么** 系统必须仅展开第一层可见节点
- **那么** 系统不得默认展开全部后代节点

#### 场景:恢复上次展开状态
- **当** 用户重新打开工作台
- **那么** 系统必须恢复上次保存的节点展开与折叠状态
- **那么** 恢复逻辑必须只依赖本地存储，而不得强制请求远端状态

### 需求:知识树节点必须展示内容状态与摘录计数
系统必须在每个知识树节点上展示 `node_title`，并根据节点数据状态显示内容标记：当 `outline_markdown` 不为空时必须显示笔记标记；当本地存在挂载到该节点的摘录时必须显示摘录数量标记，数量来源于 `excerpts.tree_node_id` 的本地统计结果。

#### 场景:节点存在大纲内容
- **当** 某个节点的 `outline_markdown` 不为空
- **那么** 系统必须在该节点上显示笔记状态标记

#### 场景:节点挂载了摘录
- **当** 某个节点下存在一条或多条挂载摘录
- **那么** 系统必须显示对应的摘录数量
- **那么** 摘录数量必须随着挂载和解除挂载即时变化

### 需求:知识树画布必须支持键盘优先的快速建树
系统必须为知识树画布提供键盘优先的创建和导航操作。`Tab` 必须用于创建当前节点的子节点，`Enter` 必须用于创建同级兄弟节点，方向键必须用于在当前树结构中切换选中节点，`Space` 必须用于切换右侧沉淀抽屉的可见性。

#### 场景:按 Tab 创建子节点
- **当** 用户在画布中选中某个节点并按下 `Tab`
- **那么** 系统必须在该节点下创建新的子节点
- **那么** 新节点必须处于可继续编辑的状态

#### 场景:按 Enter 创建同级节点
- **当** 用户在画布中选中某个节点并按下 `Enter`
- **那么** 系统必须在该节点同级位置创建新的兄弟节点
- **那么** 新节点的排序位置必须符合当前同级 `sort_order` 规则

#### 场景:使用方向键切换节点
- **当** 用户按下方向键
- **那么** 系统必须在可见节点之间切换当前选中节点
- **那么** 切换结果必须保持在当前树上下文内

#### 场景:按 Space 切换抽屉
- **当** 用户在知识树画布中按下 `Space`
- **那么** 系统必须切换右侧沉淀抽屉的可见性

### 需求:知识树画布必须支持视图缩放和平移控制
系统必须在知识树画布右下角提供视图控制入口，包括小地图、放大、缩小和适应屏幕。用户在画布内拖拽和平移视图时，当前节点选择和展开状态不得被重置。

#### 场景:使用缩放控制
- **当** 用户点击放大或缩小按钮
- **那么** 系统必须调整当前知识树画布缩放级别

#### 场景:适应屏幕
- **当** 用户点击适应屏幕按钮
- **那么** 系统必须将当前可见节点调整到可视区域内
- **那么** 系统不得改变节点的排序或展开状态

### 需求:知识树画布必须支持同级重排与稀疏排序
系统必须在知识树画布中支持节点的同级重排，并使用稀疏排序字段 `sort_order` 记录顺序。新增或移动节点时，系统必须优先使用预留间隔值分配顺序，默认步长为 `65536`，以减少频繁重排。

#### 场景:创建新同级节点
- **当** 用户在某个同级层级下创建新节点
- **那么** 系统必须为该节点分配可插入的稀疏 `sort_order`
- **那么** 系统不得重新编号整个同级列表

#### 场景:移动同级节点
- **当** 用户调整某个节点在同级列表中的位置
- **那么** 系统必须仅更新受影响节点的 `sort_order`
- **那么** 系统必须保持其他节点的稀疏间隔
