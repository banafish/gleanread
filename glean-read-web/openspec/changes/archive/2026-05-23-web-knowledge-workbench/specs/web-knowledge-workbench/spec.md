# 规范：web-knowledge-workbench

## 目的
定义 GleanRead Web 端主工作台的路由壳、全局导航和三栏页面骨架，确保用户在 Web 端能够从登录入口进入受保护工作台，并在同一页面框架内完成摘录浏览、知识树构建和右侧沉淀。

## 需求

### 需求:Web 端必须提供独立的路由壳与受保护工作区
系统必须在 Web 端提供 `/`、`/login`、`/auth/callback` 和 `/app` 四个路由，并保证它们承担各自固定职责：`/` 作为产品入口，`/login` 作为独立登录页，`/auth/callback` 作为认证回调收敛页，`/app` 作为受保护的主工作台。未完成认证的用户访问 `/app` 时，系统必须引导其进入登录流程而不是直接暴露工作台内容。

#### 场景:打开产品入口页
- **当** 用户直接访问 `/`
- **那么** 系统必须展示 GleanRead 的产品入口和开始使用入口
- **那么** 页面不得直接进入受保护的工作台内容

#### 场景:访问受保护工作区
- **当** 未认证用户直接访问 `/app`
- **那么** 系统必须将其引导到登录入口或等价的认证流程
- **那么** 系统不得在未认证状态下渲染工作台主体内容

#### 场景:处理认证回调
- **当** 用户从邮箱验证、OAuth 或其他认证跳转进入 `/auth/callback`
- **那么** 系统必须完成凭证接收、会话落盘与状态恢复
- **那么** 成功处理后系统必须重定向到 `/app`

### 需求:主工作台必须以三栏骨架承载核心知识流
系统必须在 `/app` 提供固定的三栏工作台骨架：左侧为收件箱侧栏，中间为横向知识树画布，右侧为详情沉淀抽屉。桌面端主布局必须保持左中右三栏同时可见，默认宽度目标分别约为 280px、剩余自适应空间和 400px；其中左侧侧栏与右侧抽屉可以独立折叠或展开，但中间画布始终是页面主体。

#### 场景:首次进入工作台
- **当** 用户登录后首次进入 `/app`
- **那么** 系统必须显示左侧收件箱、中间画布和右侧抽屉的工作台骨架
- **那么** 页面不得退化为单栏或跳转式多页面界面

#### 场景:切换侧栏可见性
- **当** 用户折叠左侧收件箱或展开右侧抽屉
- **那么** 系统必须仅调整对应栏位的可见性与宽度
- **那么** 中间画布的主体定位关系必须保持稳定

### 需求:顶部导航必须提供全局搜索、主题切换与账户入口
系统必须在工作台顶部提供统一导航栏，且导航栏必须包含品牌标识、全局搜索入口、主题切换入口以及账户或设置入口。全局搜索必须能从键盘快捷键 `Ctrl + K` 或 `Cmd + K` 唤出，并在关闭后返回原工作台上下文。

#### 场景:打开全局搜索
- **当** 用户点击顶部搜索入口或按下 `Ctrl + K` / `Cmd + K`
- **那么** 系统必须打开全局搜索弹窗
- **那么** 用户关闭弹窗后必须回到原来的工作台视图

#### 场景:切换主题
- **当** 用户点击顶部主题切换入口
- **那么** 系统必须在浅色和深色工作模式之间切换
- **那么** 切换结果必须立即反映到当前工作台的视觉样式

#### 场景:进入账户相关操作
- **当** 用户点击顶部账户或设置入口
- **那么** 系统必须提供账户相关操作入口
- **那么** 工作台布局不得因此被整体替换为其他页面

### 需求:工作台页面状态必须可恢复并保持上下文
系统必须在 Web 工作台内维护当前选中节点、侧栏展开状态、抽屉开关和视图位置等页面级状态，并在刷新或重新进入时尽可能恢复用户的工作上下文。左侧收件箱和右侧抽屉的内容切换不得破坏中间画布的当前浏览位置。

#### 场景:刷新后恢复工作台上下文
- **当** 用户在工作台中刷新页面
- **那么** 系统必须尽可能恢复上一次的页面级状态
- **那么** 用户不得被重置回完全无上下文的初始视图

#### 场景:切换选中节点
- **当** 用户在中间画布中切换当前选中节点
- **那么** 系统必须更新右侧抽屉所对应的内容上下文
- **那么** 左侧收件箱列表的浏览位置不得被无关重置
